# Serverless Data Lake & ETL Pipeline with AWS Glue

## 📌 Project Overview
This project demonstrates an end-to-end serverless data engineering pipeline built entirely on AWS. It ingests raw CSV data, infers the schema automatically, enables ad-hoc SQL querying without a physical database, and transforms the data into a highly optimized format using a Spark-based ETL job. 

The pipeline is fully version-controlled, with AWS Glue scripts automatically syncing to this GitHub repository.

## 🛠️ Architecture & Tech Stack
* **Amazon S3:** Centralized Data Lake utilized for raw data storage, query results, and processed data.
* **AWS IAM:** Secure role-based access control (`glue-iam`) allowing Glue services to interact with S3 and CloudWatch.
* **AWS Glue Data Catalog & Crawlers:** Automated schema discovery and metadata repository.
* **Amazon Athena:** Serverless interactive query service used to analyze S3 data using standard SQL.
* **AWS Glue ETL (Visual):** PySpark-based data transformation engine.
* **GitHub Integration:** CI/CD practice connecting AWS Glue directly to GitHub for version control.

## 🗂️ Dataset
The project utilizes a dummy dataset (`customers-100.csv`) containing sample customer records.

## 🚀 Step-by-Step Implementation

### 1. Data Ingestion & Storage (Amazon S3)
* Created a central S3 bucket named `harsh-s3-1-bucket` with the following distinct folder structure:
  * `s3://harsh-s3-1-bucket/landing_zone/` (For raw CSV ingestion)
  * `s3://harsh-s3-1-bucket/query_data/` (For storing Amazon Athena query results)
  * `s3://harsh-s3-1-bucket/transformed/` (For the final optimized output)
* Uploaded raw customer CSV data into the landing zone.

### 2. Schema Discovery (AWS Glue Crawler)
* Created a custom IAM role named `glue-iam` with `AmazonS3FullAccess`, `AWSGlueConsoleFullAccess`, and `CloudWatchLogsFullAccess` policies.
* Configured and ran an AWS Glue Crawler named `customers` targeting the S3 landing zone.
* The crawler successfully inferred the CSV schema and automatically generated a table named `customers` inside a logical Glue database named `sample_db`.

### 3. Ad-Hoc Querying (Amazon Athena)
* Connected Amazon Athena to the `sample_db` Glue Data Catalog.
* Configured Athena to save query results to the `query_data/` S3 folder.
* Executed standard SQL queries directly against the raw CSV files sitting in S3 without moving the data or spinning up a traditional relational database server.

### 4. Data Transformation (AWS Glue Visual ETL)
* Built a serverless Spark job named `customers_dropField` using the AWS Glue Visual ETL interface.
* **Source:** Extracted data from the `sample_db` catalog (the `customers` table).
* **Transformation:** Applied data cleaning steps using the "Drop Fields" node to remove unnecessary or sensitive columns.
* **Target:** Loaded the transformed DataFrame back into the `s3://harsh-s3-1-bucket/transformed/` folder, converting the output format for optimized downstream analytics.

### 5. Automation & Version Control
* Linked the AWS Glue Studio environment to this GitHub repository via a Personal Access Token (PAT).
* Pushed the underlying PySpark script generated by the `customers_dropField` Visual ETL tool directly into the repository for strict version control.
